# Shakti Design Constitution

## Status

**Master design constitution for Shakti System OS, Shakti Shala, Start Your Shakti Path, and the Shakti Shala Vault.**

This document is the reference layer every design, prototype, prompt, screen, component, and future implementation should read before making design decisions.

It extends the current `DESIGN.md` and `README.md` without replacing them.

---

# 1. Core Thesis

```text
The medicine is already real.
The system exists to protect, clarify, and steward it.
```

The design must never flatten Sheetal Kandola’s work into a generic wellness app, tantra brand, course dashboard, or conversion funnel.

The design posture is:

```text
Natural sanctuary on the surface.
Precision architecture beneath.
```

The experience posture is:

```text
Steady first.
Orient second.
Route third.
```

---

# 2. What We Are Building

Shakti Shala is not merely a website, app, or membership portal.

It is a **sanctuary operating system** for Sheetal Kandola’s ecosystem.

It contains, or will eventually contain:

- Start Your Shakti Path
- Digital self-audit / orientation
- Shakti Shala Vault
- Temple Library
- Goddess Pathways
- Daily Practice
- Moon Rhythm
- Sacred Journal
- Retreat Readiness
- Community rhythm
- Audio reflections
- Teacher / guide pathways
- Retreat companion
- Future Shakti Fest / sanctuary architecture

The product should feel like a place people return to, not software they operate.

# 3. Core Spatial Model

Do not design pages.

Design places.

The user should remember where they are emotionally and spatially.

Recommended spatial language:

```text
Temple Gates
Courtyard
Fire Circle
Temple Library
Reflection Pool
Moon Chamber
Goddess Chambers
Practice Room
Mountain Path
Inner Sanctuary
Retreat Passage
```

Default rule:

```text
People remember places, not pages.
```

---

# 4. Progressive Simplification Rule

The deeper someone travels, the less interface they should see.

```text
Temple Gates = orientation and choice
Temple Library = curated structure
Goddess Pathway = focused chamber
Single Practice = almost no interface
```

The deepest practice spaces should shed visible navigation, controls, cards, and interface chrome.

A user should feel less stimulated, not more stimulated, the longer they remain inside the system.

---

# 5. Geography And Mythic Setting

Shakti Shala is not visually or mythically a Bali product.

Bali may host retreats.

The mythic center of gravity should be Himalayan.

Draw inspiration from:

- Himalayan temples
- sacred mountains
- Kashmir Shaivism
- stone monasteries
- ancient rivers
- snow light
- moonlit paths
- sacred fire
- mountain silence
- pilgrimage

Avoid making the interface feel like:

- tropical wellness retreat
- boho Bali villa
- generic yoga travel brand
- resort booking experience

The digital sanctuary should feel timeless, high-altitude, elemental, and rooted.

---

The world should feel like:

```text
ember light on old stone

warmth inside high-altitude silence
```

Use:

- obsidian

- clay
- deep burgundy

- ash ivory
- aged bronze
- river shadow
- snow light later

Avoid:

- neon gradients
- generic lotus motifs
- psychedelic cosmic visuals
- glossy wellness illustrations
- bright SaaS dashboards
- tropical resort palettes
- loud mandalas
- tech futurism

---

# 7. Typography

Primary direction:

```text
Refined serif display + humanist sans body
```

Use serif for:

- thresholds
- sanctuary statements
- poetic-precise headers
- major transitions

Use humanist sans for:

- instructions
- body copy
- forms
- pathway descriptions
- operational clarity

Avoid default tech-mono as a core voice. Monospace accents may be used only sparingly for internal/system labels, never as the emotional voice.

---

# 8. Motion Constitution

Motion should regulate the nervous system rather than stimulate attention.

Motion should feel like:

- breath
- water
- candlelight
- incense smoke
- mountain mist
- slow arrival
- fire held steady

Use:

- slow fades
- subtle parallax
- gentle light shifts
- atmospheric grain
- breathing orbs
- soft page transitions
- quiet reveal sequences

Avoid:

- bounce
- confetti
- reward effects
- dopamine loops
- spinning loaders
- elastic animations
- aggressive parallax
- notification-style motion

A good test:

```text
The user should feel their system slow down.
```

# 10. Prompt Text Rule

Anything written in a prompt is instruction.

It is never user-facing copy.

Do not place prompt text, explanations, design philosophy, build notes, architectural commentary, or editorial instructions anywhere inside the product interface.

Only create interface copy that would naturally appear inside the product.

---

# 11. Access Model

Do not default to subscription language.

Use an initiation-based access model.

Suggested levels:

```text
Visitor
Seeker
Practitioner
Initiate
Temple Circle
Guide
Teacher
```

Access represents readiness, trust, relationship, and container—not just payment.

Some spaces may be:

- open
- member-only
- application-based
- retreat-prep only
- invitation-based
- teacher / guide only
- internal archive only

Suggested language:

```text
Initiation Key
Temple Key
Circle Access
Retreat Passage
Guide Access
```

Avoid:

- Basic / Pro / Premium
- paid tier graphics
- aggressive locked-content psychology
- scarcity pressure

---

# 12. Monetization Principle

Monetization must not compromise trust.

The system may monetize through:

- paid circles
- containers
- retreats
- courses
- audio libraries
- ceremony archives
- teacher training
- memberships
- tokenized access
- application-based intensives

But the interface should never feel extractive.

Commerce should feel like entering the right container, not being pushed through a funnel.

---

# 13. Tokenized Access Principle

Tokenized access may be used as an operational and symbolic layer.

It should feel like holding a key, not buying a speculative asset.

Do not center crypto language in the seeker-facing experience.

Public-facing language:

```text
Key
Passage
Circle
Access
Temple Key
```

Internal language:

```text
token-gated access
wallet status
membership credential
permission layer
```

Use tokenized access to:

- verify membership
- unlock content chambers
- recognize retreat participation
- manage invitation-only spaces
- preserve lineage-sensitive material
- create portable membership credentials

Do not use it to:

- financialize the medicine
- create speculative hype
- replace human discernment
- bypass application review

---

# 14. Content System Principle

The app should not become the storage dump.

Google Drive is the living content library during the early build.

GitHub is the application brain.

Airtable tracks structured readiness and access data.

Notion holds human operating clarity.

Graphify maps relationships across doctrine, files, workflows, and decisions.

```text
Google Drive = media and source assets
GitHub = code, design system, prompts, schemas
Airtable = applications, readiness, access, retreat pipeline
Notion = operating dashboard, team docs, editorial rhythm
Graphify = context graph and long-term memory
```

Do not duplicate content unnecessarily.

# 17. AI Behavior Rules

AI should support retrieval, orientation, and reflection.

AI must not:

- imitate Sheetal
- diagnose
- perform therapy
- replace human discernment
- simulate intimacy
- push users into deeper containers
- answer beyond the doctrine or allowed content

AI may:

- help a seeker find a practice
- summarize their own journal patterns back to them
- suggest relevant teachings
- remind them of their stated rhythm
- route them toward human support when appropriate

The seeker-facing AI posture should be:

```text
quiet guide, not guru
library steward, not healer
orientation layer, not authority
```

# 18. Design Review Test

Every design must pass these questions:

1. Does this feel like sanctuary, not software?
2. Does this protect Sheetal’s medicine from being flattened?
3. Does this help a woman return to practice?
4. Does the interface become quieter as she goes deeper?
5. Does this preserve readiness and human discernment?
6. Does this avoid manipulative urgency?
7. Does this feel rooted in the Himalayan mythic field, not generic wellness?
8. Does it create spatial memory?
9. Does it keep content organized without turning the system into a content dump?
10. Would someone say, “I spent time somewhere,” not “I used an app”?

- No generic wellness app design.
- No shallow tantra language.
- No Bali-as-primary-world framing.
- No SaaS dashboard front-end energy.
- No over-automation of intimacy.
- No prompt text exposed in the UI.
- No gambling/reward-style gamification.
- No overbuilding before pathway clarity.
- No content duplication across systems without reason.
- No retreat booking pressure without readiness.
